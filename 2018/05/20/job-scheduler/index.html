Android 5.0 JobScheduler全方位解析

发表于 2018-05-20 | 分类于 Android | 阅读次数:

JobScheduler是Android 5.0（API 21）引入的用于调度任务或工作的Android框架API。值得注意的是，Android 7.0（API 24）增加在ContentProvider发生改变时触发你的工作计划的功能。

特性

  • 支持在一个任务上组合多个条件
  • 内置条件：设备待机、设备充电和连接网络
  • 支持持续的job，这意味着设备重启后，之前被中断的job可以继续执行
  • 支持设置job的最后执行期限
  • 根据你的配置，可以设置job在后台运行还是在主线程中运行

应用场景

需要在Android设备满足某种场合才需要去执行处理数据：

  • 应用具有可以推迟的非面向用户的工作(定期数据库数据更新)
  • 应用具有当插入设备时希望优先执行的工作(充电时才希望执行的工作备份数据)
  • 需要访问网络或Wi-Fi连接时需要进行的任务(如向服务器拉取内置数据)
  • 希望作为一个批次定期运行的许多任务(s)

特征

1、Job Scheduler只有在Api21或以上的系统支持。
2、Job Scheduler是将多个任务打包在一个场景下执行。
3、在系统重启以后,任务会依然保留在Job Scheduler当中,因此不需要监听系统启动状态重复设定。
4、如果在一定期限内还没有满足特定执行所需情况,Job Scheduler会将这些任务加入队列,并且随后会进行执行。

用法

创建JobService

派生JobService子类，定义需要执行的任务（UI线程）。

public abstract boolean onStartJob(JobParameters params)

当开始一个任务时，onStartJob是必须使用的方法，因为它是系统用来触发已经安排的工作/任务的。从上边的用例代码可以看到，该方法返回一个布尔值。不同的返回值对应了不同的处理方式。

  • 如果返回值是false，该系统假定任何任务运行不需要很长时间并且到方法返回时已经完成
  • 如果返回值是true，那么系统假设任务是需要一些时间并且是需要在自己的应用执行的
    当给定的任务完成时需要通过调用public final void jobFinished(JobParameters params, boolean wantsReschedule)方法告知系统
    该任务已经处理完成。
    如果返回值为true，需要手动调用jobFinished来停止该任务。

public final void jobFinished (JobParameters params, boolean wantsReschedule)

回调通知已完成执行的JobManager。这可以从任何线程调用，因为它最终将在应用程序的主线程上运行。当系统收到该消息时，它将释放正在保存的唤醒。

  • 参数params：传入的param需要和onStartJob中的param一致
  • 参数wantsReschedule：如果这项工作应按照计划时间指定的停止条件进行重新安排，则传入true。否则的话传入false。就是让系统知道这个任务是否应该在最初的条件下被重复执行。

public abstract boolean onStopJob(JobParameters params)

当收到取消请求时调用，onStopJob是系统用来取消挂起的任务的。重要的是要注意到，如果onStartJob返回false，当取消请求被接收时，该系统假定没有目前运行的工作。换句话说，它根本就不调用onStopJob。那此时就需要手动调用jobFinished方法了。

注意：工作服务需要在应用程序的主线程上运行。这意味着，须使用另一个线程处理程序，或运行时间更长的任务异步任务用以不阻塞主线程。

绑定JobService任务

同Service一样，需要在AndroidManifest.xml中添加一个Service节点让应用拥有绑定和使用这个JobService的权限。

获取JobScheduler对象

构建JobInfo对象

使用JobInfo.Builder来构建一个JobInfo对象

JobInfo常量：

  • BACKOFF_POLICY_EXPONENTIAL、BACKOFF_POLICY_LINEAR
    与public JobInfo.Builder setBackoffCriteria(long initialBackoffMillis, int backoffPolicy)方法中的中backoffPolicy对应
  • DEFAULT_INITIAL_BACKOFF_MILLIS
    默认的执行延迟时间（毫秒）
  • MAX_BACKOFF_DELAY_MILLIS
    最大的执行延迟时间（毫秒）
  • NETWORK_BYTES_UNKNOWN
  • NETWORK_TYPE_ANY
    与public JobInfo.Builder setRequiredNetworkType(int networkType)方法中networkType相对应，网络状态联网就行
  • NETWORK_TYPE_NONE
    与public JobInfo.Builder setRequiredNetworkType(int networkType)方法中networkType相对应，默认的网络连接状态
  • NETWORK_TYPE_CELLULAR
  • NETWORK_TYPE_METERED
  • NETWORK_TYPE_NOT_ROAMING
    与public JobInfo.Builder setRequiredNetworkType(int networkType)方法中networkType相对应，移动网络连接情况
  • NETWORK_TYPE_UNMETERED
    与public JobInfo.Builder setRequiredNetworkType(int networkType)方法中networkType相对应，WIFI连接情况

JobInfo.Builder接收两个参数：

  • jobId：要运行的任务的标识符
  • jobService：Service组件的类名

JobInfo.Builder常用方法：

  • addTriggerContentUri(JobInfo.TriggerContentUri uri)（API 24）
    添加一个TriggerContentUri，该Uri将利用ContentObserver来监控一个Content Uri，当且仅当其发生变化时将触发任务的执行。为了持续监控content的变化，需要在最近的任务触发后再调度一个新的任务（需要注意的是触发器URI不能与setPeriodic(long)或setPersisted(boolean)组合使用。要持续监控内容更改，需要在完成JobService处理最近的更改之前，调度新的JobInfo，观察相同的URI。因为设置此属性与定期或持久化Job不兼容，这样做会在调用build()时抛出IllegalArgumentException异常。)
  • setBackoffCriteria(long initialBackoffMillis, int backoffPolicy)（API 21）
    特殊：设置回退/重试的策略。类似网络原理中的冲突退避，当一个任务的调度失败时需要重试，所采取的策略。第一个参数时第一次尝试重试的等待间隔，单位为毫秒，预设的参数有：DEFAULT_INITIAL_BACKOFF_MILLIS（30000）、MAX_BACKOFF_DELAY_MILLIS（18000000）。第二个参数是对应的退避策略，预设的参数有：BACKOFF_POLICY_EXPONENTIAL二进制退避。等待间隔呈指数增长BACKOFF_POLICY_LINEAR。
  • setClipData(ClipData clip, int grantFlags)（API 26）
    设置与此任务关联的ClipData。提供ClipData的主要目的是允许授予与剪贴（clip）关联的数据的URI权限。通过grantFlags指定要授予执行的权限的确切类型，grantFlags必须为Intent.FLAG_GRANT_READ_URI_PERMISSION, Intent.FLAG_GRANT_WRITE_URI_PERMISSION和Intent.FLAG_GRANT_PREFIX_URI_PERMISSION的组合。如果ClipData包含Intents的项，则这些Intents中的任何授权标志都将被忽略。只有作为该方法参数提供的标志才有用，并且将应用于剪贴中的所有Uri或Intent项（或剪贴的子项）。由于设置此属性与持久任务不兼容，因此在调用build()时会抛出IllegalArgumentException异常。
  • setEstimatedNetworkBytes(long downloadBytes, long uploadBytes)（API 28）
    设置此任务将执行的网络流量的估计大小（以字节为单位）。无法提供合理估算的任务应使用标记值JobInfo.NETWORK_BYTES_UNKNOWN。请注意，当设备网络连接较差时，系统可能会选择延迟那些网络使用量估计较大的任务，以节省电量。此处提供的值仅反映基础任务将执行的流量；如果使用JobWorkItem，那么还需要定义每个工作项在构建时使用的网络流量。
  • setExtras(PersistableBundle extras)（API 21）
    设置可选的附加数据。这是持久的，所以只允许原始类型。
  • setImportantWhileForeground(boolean importantWhileForeground)（API 28）
    将其设置为true表示此任务非常重要，而调度应用程序位于前台或临时白名单中以用于后台限制。这意味着系统在这段时间内会放松对这项工作的休眠限制。应用程序应将此标志仅用于短期工作，这些工作对于应用程序在前台正常运行至关重要。请注意，一旦调度应用程序不再由后台限制和后台列入白名单，或由于约束条件不满而导致任务失败，则此任务的行为与没有此标志的其他任务的行为相同。
  • setMinimumLatency(long minLatencyMillis)（API 21）
    指定任务的延迟执行时间（毫秒）。因为在定期作业上设置此属性没有意义，所以在调用build()时会抛出IllegalArgumentException异常。
  • setOverrideDeadline(long maxExecutionDelayMillis)（API 21）
    设置任务调度延迟的最后期限。即使到了规定时间时还未满足其他条件，任务也将在截止日期前执行。因为在定期作业上设置此属性没有意义，所以在调用build()时会抛出IllegalArgumentException异常。
  • setPeriodic(long intervalMillis)（API 21）
    指定该任务应该按照提供的时间间隔重复执行，每个时间段不得超过一次。无法控制在此间隔内何时执行该作业，只能保证在此间隔内最多执行一次。该函数和setMinimumLatency(long)或setOverrideDeadline(long)在构建器上一起设置将导致错误。
  • setPeriodic(long intervalMillis, long flexMillis)（API 24）
    指定该任务应该按照提供的时间间隔和弹性时间进行重复执行。该任务可以在任何时间在周期末的弹性窗口中执行。
  • setPersisted(boolean isPersisted)（API 21）
    设置是否在设备重新启动后继续执行此任务。需要RECEIVE_BOOT_COMPLETED权限。
  • setPrefetch(boolean prefetch)（API 28）
    将其设置为true表示此任务旨在预取内容，以便对该设备的特定用户的体验进行重大改进。系统可能会使用此信号来放宽最初请求的网络限制，例如允许JobInfo.NETWORK_TYPE_UNMETERED作业在计量数据可用时存在多余的计量网络上运行。系统还可以将此信号与最终用户使用模式结合使用，以确保数据在用户启动应用程序之前被预取。
  • setRequiredNetwork(NetworkRequest networkRequest)（API 28）
    设置任务所需的网络类型的详细说明。如果任务不需要网络连接，则不需要调用此方法，因为默认值为null。调用该方法将网络定义为对任务的严格条件。如果请求的网络不可用，任务永远不会运行。但setOverrideDeadline(long)可以更改此行为。调用此方法将覆盖以前由setRequiredNetworkType(int)定义的任何条件; 通常只会调用其中的一种方法。

    当任务在JobService.onStartJob(JobParameters)中执行时，请确保使用JobParameters.getNetwork()返回的特定网络，否则将使用可能不符合此限制的默认网络。

  • setRequiredNetworkType(int networkType)（API 21）
    设置任务所需的网络类型的基本描述。如果需要更精确地控制网络功能，请使用setRequiredNetwork(NetworkRequest)。如果任务不需要网络连接，则不需要调用此方法，因为默认值为JobInfo.NETWORK_TYPE_NONE。另外的可选值还有JobInfo.NETWORK_TYPE_ANY，表示需要任意一种网络才使得任务可以执行。JobInfo.NETWORK_TYPE_UNMETERED，表示设备不是蜂窝网络(比如在WIFI连接时)时任务才会被执行。还有JobInfo.NETWORK_TYPE_NOT_ROAMING和JobInfo.NETWORK_TYPE_METERED。调用该方法将网络定义为对任务的严格条件。如果请求的网络不可用，任务永远不会运行。但setOverrideDeadline(long)可以更改此行为。调用此方法将覆盖之前由setRequiredNetwork(NetworkRequest)定义的任何条件; 通常只需调用其中的一种方法。

    当任务在JobService.onStartJob(JobParameters)中执行时，请确保使用JobParameters.getNetwork()返回的特定网络，否则将使用可能不符合此限制的默认网络。

  • setRequiresBatteryNotLow(boolean batteryNotLow)（API 26）
    指定要执行此任务，设备的电池电量不得低。这默认为false。如果属实，则只有当电池电量不低时才会执行该任务，这通常是用户发出“电量不足”警告的点。
  • setRequiresCharging(boolean requiresCharging)（API 21）
    指定要执行此任务，设备必须正在充电（或者是非电池供电设备连接到永久电源，如Android TV设备）。默认为false。

    为了执行任务，电池供电的设备“充电”与简单地连接电源并不完全相同。如果设备非常繁忙，即使电源连接，电池仍然耗尽，具有此限制的作业将不会运行。这可能发生在一些常见的使用情况下，例如视频聊天，特别是如果设备插入USB而不是墙上电源。也就是说要插入充电器，而且还要在电池处于健康状态的情况下才会触发，一般来说是手机电量大于15%。

  • setRequiresDeviceIdle(boolean requiresDeviceIdle)（API 21）
    设置为true时，确保该任务在设备处于活动状态时不会运行。默认为false：即有人正在与设备进行交互，作业也可以运行。此状态是系统提供的松散定义。一般来说，这意味着该设备目前没有被交互使用，并且还没有被使用一段时间。因此，现在是执行资源繁重工作的好时机。请记住，电池使用量仍将归因于应用程序，并在电池状态中显示给用户。

    尽管名称相似，但这个任务约束与系统的“设备闲置”或“休眠”状态无关。此限制条件仅用于确定是否允许任务在设备正在使用时运行。

  • setRequiresStorageNotLow(boolean storageNotLow)（API 26）
    指定要执行此任务，设备的可用存储空间不得低。默认为false。如果设置为true，则只有在设备未处于低存储状态时才会执行任务，这通常是用户获得“低存储量”警告的关键点。
  • setTransientExtras(Bundle extras)（API 26）
    设置可选的短暂的附加数据。由于设置此属性与持久作业不兼容，因此在调用build()时会抛出IllegalArgumentException异常。
  • setTriggerContentMaxDelay(long durationMs)（API 24）
    设置从content变化到任务被执行中间的最大延迟。（以毫秒为单位）。
  • setTriggerContentUpdateDelay(long durationMs)（API 24）
    设置从content变化到任务被执行中间的延迟。如果在延迟期间content发生了变化，延迟将被重置为在最近一次变化时开始。

开启任务

在调用schedule方法时，会返回一个int类型的值来标记这次任务是否执行成功，如果返回小于0的错误码，这表示该次任务执行失败，反之则成功(成功会返回该任务的ID)。如果需要停止一个任务，就通过JobScheduler中cancel(int jobId)来实现；如果想取消所有的任务，可以调用JobScheduler对象的cancelAll()来实现。
