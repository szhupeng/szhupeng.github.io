AIDL跨进程通信分析

发表于 2017-03-18 | 分类于 Android | 阅读次数:

在Android中，每一个进程都有单独的Dalvik虚拟机实例，都有一块独立的内存，并且在这块独立的内存上存储数据，执行操作。两个进程无法直接通信，必须通过Android系统底层间接通信。而AIDL就是这种实现底层通信的方式。

Note: Using AIDL is necessary only if you allow clients from different applications to access your service for IPC and want to handle multithreading in your service. If you do not need to perform concurrent IPC across different applications, you should create your interface by implementing a Binder or, if you want to perform IPC, but do not need to handle multithreading, implement your interface using a Messenger.

只有允许不同应用的客户端用IPC方式访问服务，并且想要在服务中处理多线程时，才有必要使用AIDL。如果不需要执行跨应用的并发IPC，就应该通过实现一个Binder创建接口；或者，如果想执行IPC，但根本不需要处理多线程，则使用Messenger类来实现接口。

AIDL接口的调用是直接函数调用。不应该猜测发生调用的线程。根据调用是来自本地进程还是远程进程中的线程，实际情况会有所差异。具体而言：

  • 从本地进程发起的调用在进行调用的同一线程内执行。如果该线程是主线程，则该线程继续在AIDL接口中执行。如果该线程是其他线程，则其便是在服务中执行代码的线程。因此，只有在本地线程访问服务时，才能完全控制哪些线程在服务中执行（但如果真是这种情况，就不应该使用AIDL，而应该通过实现Binder类创建接口）。
  • 来自远程进程的调用从平台在你自己的进程内部维护的线程池中发出。必须为来自未知线程的多次并发调用做好准备。换言之，AIDL接口的实现必须是完全线程安全的。
  • oneway关键字用于改变远程调用的行为。使用该关键字时，远程调用不会阻塞；它只是发送事务数据并立即返回。接口的实现最终接收此调用时，是以正常远程调用形式将其作为来自Binder线程池的常规调用进行接收。如果oneway用于本地调用，则不会有任何影响，调用仍是同步的。

从代码可以看到客户端最终是通过这个Add.Stub.Proxy类和服务端进行通信的。

  • 方法的传参的数据会存入_data中，而将方法的返回值的数据会存入_reply中。如果涉及到in，out，inout这三个定向tag，会有所不同。

    All non-primitive parameters require a directional tag indicating which way the data goes. Either in, out, or inout. Primitives are in by default, and connot be otherwise.

    意思就是：所有的非基本参数都需要一个定向tag来指出数据流通的方式，不管是in, out, 还是inout。基本参数的定向tag默认是并且只能是in。
    其实AIDL中的定向tag表示了在跨进程通信中数据的流向：

    1. in表示数据只能由客户端流向服务端
    2. out表示数据只能由服务端流向客户端
    3. inout 则表示数据可在服务端与客户端之间双向流通

    其中，数据流向是针对在客户端中的那个传入方法的对象而言的。

    1. in为定向tag的话表现为服务端将会接收到一个那个对象的完整数据，但是客户端的那个对象不会因为服务端对传参的修改而发生变动
    2. out的话表现为服务端将会接收到那个对象的的空对象，但是在服务端对接收到的空对象有任何修改之后客户端将会同步变动
    3. inout为定向tag的情况下，服务端将会接收到客户端传来对象的完整信息，并且客户端将会同步服务端对该对象的任何变动
  • Parcel是一个用来存放和读取数据的容器。可以用它来进行客户端和服务端之间的数据传输，当然，它能传输的只能是可序列化的数据。
  • transact()方法是客户端和服务端通信的核心方法。调用这个方法之后，客户端将会挂起当前线程，等候服务端执行完相关任务后通知并接收返回的_reply数据流。
    • 方法ID：transact()方法的第一个参数是一个方法ID，这个是客户端与服务端约定好的给方法的编码，彼此一一对应。在AIDL文件转化为.java文件的时候，系统将会自动给AIDL文件里面的每一个方法自动分配一个方法ID。
    • transact()方法的第四个参数是一个int值，它的作用是设置进行IPC的模式，为0表示数据可以双向流通，即_reply流可以正常的携带数据回来，如果为1的话那么数据将只能单向流通，从服务端回来的_reply流将不携带任何数据。AIDL生成的.java文件的这个参数均为0。

从源码可以看出来，Proxy类的方法里面的工作流程：

  1. 生成_data和_reply数据流，并向_data中存入客户端的数据
  2. 通过transact()方法将它们传递给服务端，并请求服务端调用指定方法
  3. 接收_reply数据流，并从中取出服务端传回来的数据

客户端调用transact()方法将数据和请求发送到服务端以后，服务端有一个方法来接收这些传过来的东西：在Add.java里面可以找到一个叫做onTransact()的方法。

当服务器接收了客户端的transact()方法传过来的参数后，根据方法ID进入对应的switch分支。直接调用服务端这边的具体方法实现，然后获取返回值并将其写入reply流中。而reply回传到客户端的细节被隐藏了，这样transact()和onTransact()之间的调用以及数据传送看起来就像是发生在同一个进程甚至同一个类里面一样。

服务端的工作流程：

  1. 获取客户端传过来的数据，根据方法 ID 执行相应操作
  2. 将传过来的数据取出来，调用本地写好的对应方法
  3. 将需要回传的数据写入reply流，传回客户端

设计思想

通过代理-桩的设计理念屏蔽内部细节。客户端将需求提供给代理，代理收到需求以后就去找桩，桩拿到需求让服务器执行相应事件。执行结束后通过桩将结果告诉代理，代理再把结果告诉客户端。这样客户端和服务端像在一个进程中一样。
